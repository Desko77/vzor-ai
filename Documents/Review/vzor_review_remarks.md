# Vzor — Ревью архитектурных документов

**Дата ревью:** 2026-03-04  
**Ревьюер:** Claude (по запросу Дмитрия)  
**Документы:**
- vzor_architecture_overview.md

- vzor_runtime_sequences.md

**Общая оценка:** сильная архитектурная проработка для стадии до первого кода. Чёткое разделение на 4 тира, event-driven vision pipeline с кэшем, FSM для голосового оркестратора, продуманный fallback-маршрут — всё это правильные решения для edge-AI системы.

---

## 1. Архитектура в целом

### 1.1 Отсутствует Security / Privacy слой

**Серьёзность:** Высокая

Камера очков непрерывно снимает окружающих людей — это чувствительная тема и юридически, и технически. Нигде не описано: где хранятся фреймы, как долго, шифруется ли канал Phone↔X2, есть ли consent-механизм. Для продукта с камерой это must-have раздел, даже если пока прототип.

> **Примечание:** В open_questions раздел 16 (Privacy Model) частично покрывает этот вопрос, но он появился позже основных архитектурных документов и не интегрирован в overview/diagram.

### 1.2 Нет полноценного Offline-сценария

**Серьёзность:** Средняя

BackendRouter упоминает "offline", но в диаграммах последовательностей (runtime_sequences.md) offline-путь не показан. Вопросы без ответа:
- Что конкретно работает без Wi-Fi и без LTE?
- Какая модель STT на телефоне в offline?
- Какой LLM fallback?

**Рекомендация:** добавить отдельную sequence-диаграмму для offline-сценария.

### 1.3 Нет модели данных / схемы Scene JSON

**Серьёзность:** Высокая

Scene JSON v2 упоминается многократно, но его схема нигде не зафиксирована формально. В open_questions есть пример JSON, но нет `.proto` файла или JSON Schema.

**Рекомендация:** для Sprint 1 зафиксировать хотя бы JSON Schema с обязательными полями. Раз уже выбран gRPC — сделать `.proto` файл, от которого автогенерируются Kotlin data classes и Python dataclasses.

### 1.4 Нет описания Conversation Memory / Context Window

**Серьёзность:** Средняя

Как LLM получает историю диалога? Где она хранится — на Phone, на X2? Какой лимит? Это критично для качества ответов.

> **Частично** покрыто в open_questions раздел 4 (Context Manager) и ConversationSession, но не отражено в основных архитектурных документах.

## 2. VoiceOrchestrator FSM

### 2.1 Неопределённый механизм SPEECH_END

**Серьёзность:** Средняя

Переход LISTENING → PROCESSING по SPEECH_END — а кто определяет SPEECH_END? VAD на телефоне, STT endpoint detection, или таймаут? Это разные механизмы с разными задержками. Стоит уточнить в Event Model.

### 2.2 Нет состояния CONFIRMING

**Серьёзность:** Средняя

В сценарии "Позвони маме" после GENERATING должен быть переход в состояние ожидания подтверждения. Сейчас ConfirmUI упоминается в sequence-диаграмме, но FSM это состояние не отражает.

**Рекомендация:** добавить `CONFIRMING` как отдельное состояние FSM или явно описать как ConfirmUI взаимодействует с существующими состояниями.

### 2.3 ERROR → IDLE без обратной связи

**Серьёзность:** Низкая

ERROR → IDLE через 3 секунды — а пользователь узнает об ошибке? Нужен аудио-фидбек (звук или фраза "Произошла ошибка, попробуйте снова").

> **Частично решено:** в open_questions раздел 15.3 описано поведение при ERROR, включая UI badge. Но аудио-фидбек через очки не упоминается — пользователь может не смотреть на экран телефона.

## 3. Vision Pipeline

### 3.1 Не описаны триггеры переключения FrameSampler

**Серьёзность:** Средняя

FrameSampler: адаптивный fps (1 / 5–10 / 15 + bursts) — хорошо, но кто решает перейти с 1 fps на 15? EventBuilder? VAD? Запрос пользователя? Это ключевая логика, и без неё реализация будет ad-hoc.

> **Решено в open_questions раздел 14.4** — триггеры переключения описаны. Нужно перенести в основные архитектурные документы.

### 3.2 Vision Router: TTL не определён

**Серьёзность:** Средняя

Что если пользователь спрашивает "что это?" и кэш содержит сцену 30-секундной давности? TTL по типам данных упоминается в risk map, но конкретные значения и политики не были определены в overview.

> **Решено в open_questions раздел 14.8** — дифференцированный TTL (objects: 1s, text: 3s, scene: 5s, location: 30s). Нужно отразить в основных документах.

### 3.3 MediaPipe + ML Kit OCR: параллельно или последовательно?

**Серьёзность:** Низкая

Для бюджета батареи существенно — запускаются ли MediaPipe и ML Kit OCR параллельно на каждом кадре или последовательно. Не описано.

## 4. Risk Map

### 4.1 Нет риска по батарее телефона

**Серьёзность:** Высокая

Постоянная работа камеры + BT + MediaPipe + сеть — телефон сядет быстро. Это High probability / High impact, и должен быть в топ-3 рисков.

**Рекомендация:** добавить риск "Phone battery drain" с митигацией через Adaptive Frame Sampling и Low battery mode.

### 4.2 Нет риска по DAT SDK availability

**Серьёзность:** Высокая

Meta может изменить API, ограничить доступ, или SDK просто не будет работать стабильно на конкретных устройствах. Это внешняя зависимость, которую вы не контролируете.

**Рекомендация:** добавить риск с митигацией: абстрактный SensorIngest интерфейс, позволяющий заменить DAT SDK без переписывания всего pipeline.

### 4.3 Риск #3 — слишком общая митигация

**Серьёзность:** Низкая

Митигация "implement gRPC server with priority queue" — какие приоритеты? STT > LLM > Vision? Или зависит от контекста?

> **Решено в open_questions раздел 13.8** — приоритетная матрица определена (Voice LLM > STT > Vision > YOLO > CLIP). Нужно обновить risk map.

## 5. Open Questions (vzor_open_questions.md)

### 5.1 Структурные замечания

#### Документ перерос формат "Open Questions"

**Серьёзность:** Средняя

Название подразумевает список вопросов, а фактически это полноценный Architecture Decision Record + Technical Design Document (~1000 строк). Решённые вопросы (✅) и открытые (❓) перемешаны с принятыми решениями.

**Рекомендация:** переименовать (например, "Vzor Technical Architecture") и выделить open questions в отдельный трекер или таблицу в конце.

#### Значительное дублирование между документами

**Серьёзность:** Средняя

FSM-диаграмма присутствует в 4 файлах, Vision pipeline — в 3, Sequence-диаграммы — в 2. Это создаёт риск рассинхронизации. Уже есть расхождения:
- В architecture_overview VLM указан как "Qwen-VL"
- В open_questions — "Qwen3.5 нативный multimodal"

**Рекомендация:** консолидировать до 2 документов (см. раздел 8 «Общие рекомендации»).

#### Нумерация разделов скачет

**Серьёзность:** Косметическая

Нет раздела 18, после 17 идёт 19.

### 5.2 Содержательные замечания

#### Context Manager: 20 реплик rolling window (раздел 4)

**Серьёзность:** Средняя

20 реплик — это мало для сложных разговоров, но может быть много для маленьких моделей по токенам. Стоит указать не количество реплик, а токенный бюджет (например, 2048 токенов) и суммаризацию при превышении.

Промпт "есть ли что-то что стоит запомнить?" на каждую сессию — это дополнительный LLM вызов. При 50+ сессиях в день это ощутимо. Стоит вызывать только если сессия длиннее N реплик.

#### BackendRouter: нет обработки "X2 выключен" (раздел 5.1a)

**Серьёзность:** Средняя

Алгоритм хороший, но `x2_queue_wait_ms < 800ms` подразумевает что X2 отвечает на health check. Что если X2 просто недоступен?

**Рекомендация:** добавить отдельный `x2_available` flag с heartbeat (например, ping каждые 5 секунд).

#### Синхронный перевод: BT HFP mic для собеседника (раздел 11)

**Серьёзность:** Высокая

Сценарий A (слушаю собеседника) подразумевает что BT HFP микрофон записывает собеседника. На практике BT HFP mic на очках имеет узкую направленность — он оптимизирован для голоса носителя, не собеседника напротив. Это может быть blocker для сценария A.

**Рекомендация:** добавить как открытый вопрос или тест в Sprint 1: записать речь собеседника через mic очков на расстоянии 1–2 метра, оценить качество STT.

#### ModelRegistry: cold start vs warm TTFT (раздел 13.7a)

**Серьёзность:** Низкая

Qwen3.5-9B latency "~800ms TTFT" — это на 4-bit quant? На EVO X2 с LPDDR5X-8000 это может быть оптимистично для первого запроса (cold start). Стоит разделить cold start vs warm TTFT в таблице.

#### Scene JSON: язык scene_summary (раздел 14.3)

**Серьёзность:** Средняя

`scene_summary` генерируется Qwen-VL — на каком языке? Если на английском, а LLM отвечает по-русски, это лишний "перевод" в промпте. Если на русском — Qwen-VL может хуже описывать.

**Рекомендация:** зафиксировать языковую политику. Вероятно, английский для scene_summary (лучшее качество vision-моделей) + явная инструкция LLM "ответь по-русски".

#### Error Handling: Low battery слишком агрессивен (раздел 15.4)

**Серьёзность:** Низкая

Feature flags таблица отключает vision phone-only полностью при low battery. Это может быть слишком агрессивно — 1fps с ML Kit OCR потребляет мало.

**Рекомендация:** оставить OCR в low battery режиме, отключить только MediaPipe (face/hand/pose).

#### Privacy: нет упоминания 152-ФЗ (раздел 16)

**Серьёзность:** Низкая (для pet project) / Высокая (для продукта)

Если транскрипты уходят в Yandex SpeechKit — это обработка персональных данных. Yandex как оператор? Нужно ли согласие? Для pet project не критично, но при росте стоит зафиксировать.

#### Resource Budget: модели loaded simultaneously (раздел 17.4)

**Серьёзность:** Низкая

EVO X2 итого ~14GB из 128GB — это для inference. Но Ollama держит модели загруженными в RAM. Если одновременно загружены Qwen3.5-9B + Qwen-VL + Whisper + YOLO — это ~13GB постоянно. С учётом OS и runtime — ~20GB. Всё равно ок для 128GB, но стоит уточнить "loaded simultaneously" vs "on-demand loading" и зафиксировать стратегию (keep_alive / preload).

## 6. UI Prototype

### 6.1 Отсутствующие экраны

#### Экран перевода

**Серьёзность:** Средняя

Раздел 11 open_questions описывает полноценный translation mode с push-to-talk, субтитрами оригинал+перевод, переключателем A/B/C сценариев. В прототипе этого экрана нет.

**Рекомендация:** добавить wireframe с кнопкой PTT и двумя полосами текста (оригинал/перевод). Добавить 5-й таб в bottom navigation с иконкой 🌐.

#### ConfirmUI (диалог подтверждения действий)

**Серьёзность:** Средняя

В архитектуре это ключевой safety-паттерн ("Позвонить маме?"), но в прототипе его нет.

**Рекомендация:** добавить bottom sheet или dialog с текстом действия и кнопками Да/Отмена.

### 6.2 Voice State Indicator не привязан к FSM

**Серьёзность:** Средняя

В архитектуре описаны 6 состояний FSM с визуальной индикацией (пульсирующий индикатор для IDLE, waveform для LISTENING, спиннер "Думаю..." для PROCESSING, и т.д.). В прототипе есть waveform на conversation screen, но нет явной привязки ко всем состояниям.

**Рекомендация:** показать все 6 состояний визуально, хотя бы как переключаемые варианты в прототипе.

### 6.3 Нет индикации режима роутинга

**Серьёзность:** Средняя

На Home screen нет индикации текущего режима роутинга (Wi-Fi/LTE/Offline). В архитектуре это решающий контекст — пользователь должен видеть через какой backend идут запросы.

**Рекомендация:** добавить chip или badge на Home screen: "🏠 Дома · AI Max" / "📡 LTE · Claude" / "✈️ Офлайн".

### 6.4 Нет переключателя языка в Settings

**Серьёзность:** Низкая

В разделе 10 (open_questions) описан механизм RU/EN с подтверждением голосом. В Settings нет соответствующего пункта.

### 6.5 Нет индикации батареи очков

**Серьёзность:** Средняя

На Home screen показан статус "Connected", но нет уровня заряда очков. Для пользователя это критично — очки разряжаются за ~4 часа активного использования.

### 6.6 Logs screen — developer mode

**Серьёзность:** Низкая

Logs screen отличная идея для разработки, но в production это не нужно пользователю.

**Рекомендация:** спрятать за developer mode (тройное нажатие на версию приложения в Settings, аналогично Android).

### 6.7 FAB назначение неясно

**Серьёзность:** Низкая

Floating Action Button на Home screen — для чего он? Если это активация ассистента, то дублирует wake word и кнопку на очках.

**Рекомендация:** сделать контекстным: на Home — старт разговора, на Conversation — barge-in/stop.

## 8. Общие рекомендации

### 8.1 Консолидировать документы

Сейчас 6 файлов с существенным дублированием. Рекомендация — оставить два:
1. **"Vzor Architecture"** — единый source of truth: диаграммы, FSM, contracts, decisions
2. **"Vzor Risk & Open Questions"** — живой трекер рисков и нерешённых вопросов

Остальные файлы (overview, diagram_v2, runtime_sequences) — либо интегрировать в основной, либо генерировать из него.

### 8.2 Зафиксировать Scene JSON Schema формально

Сделать `.proto` файл (раз уже gRPC) или JSON Schema в репозитории. Автогенерация Kotlin data classes и Python dataclasses из контракта предотвратит рассинхронизацию Phone ↔ X2.

### 8.3 Ввести ADR как отдельный формат

Решения сейчас помечены "ПРИНЯТО" в тексте, но без единого формата. ADR-ARCH-001 упоминается, но формат не выдержан. Рекомендуемая структура:
- Номер, дата, статус
- Контекст и проблема
- Рассмотренные альтернативы
- Решение и обоснование
- Последствия

### 8.4 Добавить раздел "What we don't build"

Явно зафиксировать scope MVP:
- Нет face recognition
- Нет multi-user
- Нет cloud storage видео
- Нет Siri/Google Assistant интеграции
- Нет ...

Это помогает держать фокус и отвечать на вопросы "а почему не X?".

### 8.5 Тестовая стратегия

Отсутствует полностью. Decision gates хорошие, но нет описания как тестировать:
- Unit tests для FSM переходов?
- Integration tests для Phone ↔ X2?
- Нагрузочное тестирование очереди на X2?
- Benchmark scripts для STT/LLM latency?

Хотя бы принципы стоит зафиксировать.

### 8.6 Основной риск проекта

Основной риск сейчас — не в архитектуре, а в **DAT SDK** (внешняя зависимость от Meta) и в том что **документация может обогнать реализацию**. Стоит как можно быстрее выйти на работающий Sprint 1 прототип и валидировать ключевые предположения на реальном железе.
